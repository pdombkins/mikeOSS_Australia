## C078 — Cell-level refresh in review tables (smallest; mostly gap-closing)

**Already exists:** `POST /tabular-review/:id/regenerate-cell` (tabular.ts:619) + frontend handler in `TabularReviewView.tsx` (regenerate button on expanded cell). But it predates Tabular v2 and lags the `/generate` pipeline.

**Gaps to close:**
1. **Typed columns (C015)** — regenerate calls `queryTabularCell()` with only `format`/`tags`; `/generate` adds `COLUMN_TYPE_HINTS[col.type]`. Fix: pass the type hint.
2. **Reference documents (C031)** — regenerate ignores `column.reference_document_id`. Fix: extract the ref doc (same access-filtered path as `/generate`, factored into a shared `loadReferenceText(db, col, userId)` helper) and include it in the prompt.
3. **Provenance (C032)** — manual edits store `edited_by` + preserve `ai_content`. Regenerating a manually-edited cell currently silently clobbers the edit. Fix: on regenerate, clear `edited_by` (value is AI-authored again), move the prior content into `previous_content` (new nullable jsonb column), and have the UI confirm before regenerating an edited cell ("This cell was manually edited — regenerating replaces your edit").
4. **Audit + cost** — add `recordAudit` (action `tabular_cell_regenerate`) and ensure the LLM call records to `query_costs` with source `tabular` (align with whatever `/generate` records; add if missing there too).

**Migration:** `20260722_01_tabular_cell_previous_content.sql` (add `previous_content jsonb` to `tabular_cells`).
**Effort:** small. No new UI surface — improves the existing button.

## C079 — Bulk CSV import: Clauses + Playbook rules

Harvey ships CSV import for Library prompts; Mike's analogue is the two curated stores where a lecturer would seed a course's worth of content: **clauses** (C026) and **playbook rules** (C002).

**Backend:**
- `POST /clauses/import` — body: CSV text (or parsed rows), max 500 rows. Columns: `title*, body*, agreement_type, guidance, tags` (tags `;`-separated), optional `project_id` applied to all. Per-row validation → response `{ imported, skipped: [{row, reason}] }`. Exact-duplicate skip (same title+body for owner).
- `POST /playbooks/:id/rules/import` — columns: `topic*, preferred, acceptable_fallback, dealbreaker, severity(low|medium|high), notes`; `position` auto-appended. Owner-only (same check as existing rule upsert).
- Embeddings for clauses batched via existing `embedText`, spend recorded via existing `recordEmbeddingCost` (source `kb_embedding`). Failures degrade gracefully (saved without embedding, same as `saveClause`).
- CSV parsing server-side with a tiny dependency-free parser (or SheetJS already used by spreadsheet.ts — whichever is already in the tree; no new deps).
- `recordAudit` action `bulk_import` with counts.

**Frontend:** "Import CSV" button on `/clauses` and on the playbook detail view; downloadable header-template; result modal listing skipped rows. 

**Migrations:** none.
**Effort:** small-medium.

## C077 — Consumption metering (per-user / per-project usage)

Legora moved Agent Pro to consumption pricing; Mike's teaching analogue is making consumption *visible* (students learn cost-aware AI use). No billing, no hard blocks.

**1. Per-project attribution (migration):** `20260722_02_query_costs_project.sql` — add nullable `project_id uuid` to `query_costs` + index. Populate at write sites that know the project: chat routes (`chat_id`→project), agent executor (`agent_step`), tabular generate/regenerate, tabularAsk, exports. KB-embedding and other project-less calls stay null. Backfill: none (historic rows stay null).

**2. User-facing usage:** `GET /user/usage?months=6` — the caller's `query_costs` aggregated by month × source × model (+ AUD totals). New **Account → Usage** page: monthly AUD chart, per-feature breakdown, this-month total. Reuses admin-analytics aggregation shapes (admin.ts:422+).

**3. Project consumption:** `GET /projects/:id/usage` (owner/editor via project_members) — spend for that matter. Small "Spend" card on the project page.

**4. Soft budgets (optional — see approval question):** `user_profiles.monthly_budget_aud` (nullable, admin-set or self-set) + `app_settings.default_monthly_budget_aud`. Nightly check (piggybacks the regwatch-style timer): >80% → notification (kind `system`); >100% → banner on assistant page. Never blocks requests.

**Admin:** analytics page gains a "by project" table (top-N projects by spend) — trivial once project_id exists.
**Effort:** medium.

## C076 — Lists: tasks, facts & deadlines on matters (largest; new primitive)

Legora's Lists = structured work items living inside the aOS, assignable and agent-executable. Mike's version attaches to **projects** (matters) and plugs into the agent runtime, notifications, and RBAC that already exist.

**Data (migration `20260722_03_list_items.sql`):**
```
list_items: id, project_id → projects, created_by, kind ('task'|'fact'|'deadline'),
  title, detail, due_at timestamptz null, status ('open'|'in_progress'|'done'|'dismissed'),
  assignee_user_id null, document_id null (fact/deadline source doc),
  citation text null (fact — free-text MNC/AGLC4 ref, never auto-fetched),
  agent_run_id null, position int, created_at, updated_at
```
RLS deny-by-default via `project_members` (owner/editor write; reviewer/viewer read) — same pattern as P3.

**Routes:** `backend/src/routes/lists.ts` mounted at `/projects/:projectId/list` — GET (grouped), POST, PATCH `/:itemId` (status/assign/edit/reorder), DELETE. All `recordAudit`-ed.

**Agent execution (the differentiator):** "Run with agent" on a task item → seeds an agent run (existing kind `assistant`, project-scoped context = item title+detail) via the existing planner → **approval gate (C030) applies unchanged** → on completion: `agent_run_id` linked, notification sent, item flagged "agent output ready" (human marks done — the agent never closes its own task; C024-style human-in-the-loop).

**Chat/agent tools** (`chat/tools/listTools.ts`): `add_list_item`, `list_list_items`, `update_list_item_status`. Registered in toolDispatcher + audit; added to ROLE_TOOLSETS: intake+drafting get write, research/review/verify get read-only (`list_list_items`) — consistent with WRITE_TOOLS confinement.

**Deadline reminders:** daily timer in index.ts (regwatch pattern, `LISTS_REMINDERS_DISABLED=1` escape hatch): items with `due_at` within 72h and status open → notify assignee (new NotificationKind `deadline`, added to the P2 union + email template).

**Frontend:** "Lists" tab on the project page — three grouped sections (Tasks / Facts / Deadlines), inline add, status toggles, due-date badges (red when overdue), assignee picker fed from project_members, "Run with agent" on tasks linking to `/agents/<runId>`. Sidebar badge count of overdue+due-soon items reuses the notifications badge pattern.

**Effort:** large (migration + routes + tools + timer + UI tab).

## scan.mjs `--no-fetch`

`node scripts/competitor-scan/scan.mjs --no-fetch` → skips vendor index fetching (no "Could not fetch …" scanNotes), **and takes no register side effects**: no `scanCount` bump, no new→seen aging, no seenUrls writes — it purely regenerates `reports/latest.html|md` from the register as-is. Usage line updated. This is what Claude-driven refresh runs will use from the sandbox (fixes the warning noise you saw, and stops the aging race that clobbered this week's NEW flags).
